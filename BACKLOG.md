# BACKLOG (идеи на потом)

Этот файл — “памятка”, чтобы мы не забыли важные улучшения, но не усложняли проект раньше времени.

## UX: расписание (отображение занятых ячеек)

### Проблема

На **компьютере** и **мобильных** сейчас не нравится визуализация **непустых ячеек** расписания (карточки записей): читаемость, размер, налезание текста, плотность, поведение при нескольких записях в один слот.

### Идеи решения (когда займёмся)

- Сделать карточку “как бейдж” + аккуратные отступы, чтобы не выглядело “кляксой”.
- Ограничить текст в карточке (`line-clamp`/ellipsis), а детали показывать в модалке.
- Для нескольких записей в один слот: показывать “+N” или стек/список внутри ячейки.
- Адаптив: на мобильном — более крупная сетка, горизонтальный скролл, фиксированный первый столбец (время) и/или фиксированная шапка площадок.
- Проверить контраст и цвета для “моя запись / чужая запись”.

## Этап 2 (SaaS): мульти‑клубность (tenant) + тарифы

### Цель

Сделать так, чтобы один и тот же Horseclub мог обслуживать **неограниченное количество клубов**, но с **разными тарифами** и **разными лимитами**, и чтобы данные клубов **не смешивались**.

### Почему не делаем это сейчас

- Мульти‑клубность меняет почти все таблицы и запросы.
- Сначала выгоднее полностью “отточить” основной функционал для одного клуба и понять реальные процессы.

---

## План внедрения (порядок важен)

### 1) Модель данных (PostgreSQL)

- **`clubs`**: клубы/организации.
  - поля: `id`, `name`, `created_at`
- **`plans`**: тарифы (лимиты и флаги функций).
  - поля (пример): `id`, `code`, `name`, `max_users`, `max_horses`, `max_trainers`, `chat_enabled`, `created_at`
- **`clubs.plan_id`** → ссылка на тариф
- Добавить `club_id` в основные сущности:
  - `users.club_id`
  - `horses.club_id`
  - `bookings.club_id`
  - `channels.club_id`, `messages` через `channel_id`
  - `invites.club_id`

Важно: везде добавить индексы по `club_id` (например, `CREATE INDEX ... ON bookings (club_id, booking_date)`).

### 2) “Текущий клуб” в запросах

- После логина пользователь должен “знать” свой `club_id` (в JWT и/или в базе).
- В каждом API‑запросе **обязательно** фильтровать по `club_id`.

Принцип: *никаких* запросов типа `SELECT * FROM bookings WHERE booking_date = ...` без `club_id`.

### 3) Миграция с “1 клуб” на “много”

- Создать один клуб “по умолчанию” (например `Default club`).
- Присвоить `club_id` всем существующим данным.
- После миграции сделать `club_id NOT NULL`.

### 4) Ограничения тарифов (лимиты)

Где проверять:

- **создание пользователей** (админка): считать пользователей в клубе, сравнить с `plans.max_users`
- **создание лошадей**: считать лошадей, сравнить с `plans.max_horses`
- **чат**: если `plans.chat_enabled = false` — скрыть ссылку на чат и блокировать API

Как возвращать ошибку:

- `409` или `403` с понятным текстом, например: “Достигнут лимит тарифа: максимум 5 пользователей”.

### 5) UI/админка клуба

- Страница “Настройки клуба”:
  - название клуба
  - текущий тариф + лимиты
  - кнопка “Управление пользователями” (то, что уже есть)

### 6) Подготовка к оплатам (не сразу)

- Хранить `billing_status`, `paid_until`, `provider_customer_id` (Stripe/YooKassa и т.п.)
- Включать/отключать функции по статусу оплаты

## Риски (о чём помнить)

- **Безопасность**: главная опасность — забыть `club_id` в одном из запросов и “дать увидеть чужие данные”.
- **Тесты**: после мульти‑клубности нужны минимальные тесты на изоляцию (user A не видит data B).

## Минимальная версия “multi‑club” (MVP)

Если делать максимально просто:

- `clubs`, `plans`, `users.club_id`
- лимит `max_users`
- фильтрация по `club_id` в `admin/users`, `bookings`

Это уже позволит продавать “много клубов” и включать разные тарифы.
